Stage 0 — Prerequisites & Dependencies
Before writing any project code, make sure your Python environment has every dependency installed and verified. Getting the install right prevents cryptic import errors mid-training. All packages listed here are open-source and CPU-compatible.
📦 Dependencies & Prerequisites
#0
▸
Python 3.10+
Required for modern type-hint syntax used across the project. Verify with python --version. Use pyenv or conda to manage versions. Python 3.11 is the recommended sweet spot for speed and compatibility.
▾
▸
gymnasium — RL environment API
The modern OpenAI Gym successor. Provides the Env base class, Box/Discrete spaces, and check_env() validator. Install: pip install gymnasium. Target version ≥ 0.29 which uses the separated terminated/truncated API required by SB3.
▾
▸
numpy — numerical core
Used everywhere: trig (cos/sin), array clipping, random sampling, and observation stacking. Install: pip install numpy. Version ≥ 1.24 recommended. NumPy is the backbone of both the environment dynamics and evaluation metrics.
▾
▸
torch — neural network backend
PyTorch backs all SB3 policy networks. CPU-only install: pip install torch --index-url https://download.pytorch.org/whl/cpu. Version ≥ 2.0 is preferred. No GPU needed for this project — the environment is lightweight enough for fast CPU training.
▾
▸
stable-baselines3 — RL algorithms
Provides PPO, SAC, TD3 with a unified Gym API. Install: pip install stable-baselines3[extra]. The [extra] flag pulls tensorboard integration, VecNormalize, and EvalCallback. Use version ≥ 2.0 for Gymnasium 0.26+ compatibility.
▾
▸
matplotlib — plots & animation
Used for training curves and arm animation (FuncAnimation). Install: pip install matplotlib. For saving .mp4 files, also install ffmpeg at the system level. Version ≥ 3.7 recommended for smoother animation APIs.
▾
▸
pyyaml — config file parsing
Reads ppo.yaml and sac.yaml hyperparameter configs at runtime so you never hardcode values. Install: pip install pyyaml. Use yaml.safe_load() — never yaml.load() which can execute arbitrary code.
▾
▸
tensorboard — training monitor
SB3 writes logs automatically. View live with tensorboard --logdir ./logs. Install: pip install tensorboard. Essential for spotting reward collapse, entropy drops, or diverging value loss early — before the full training run completes.
▾
▸
ffmpeg — video export (system)
System-level dependency for .mp4 animation export. Ubuntu: sudo apt install ffmpeg. macOS: brew install ffmpeg. Windows: use winget or chocolatey. Alternatively use pillow writer for GIF output with no system dependency.
▾
▸
pip install -r requirements.txt
Create a pinned requirements.txt: gymnasium>=0.29, numpy>=1.24, torch>=2.0, stable-baselines3[extra]>=2.0, matplotlib>=3.7, pyyaml>=6.0, tensorboard>=2.13. One command installs everything reproducibly on any machine.
▾
Stage 1 — Project Scaffold & Config
Establish the directory structure and shared utilities every other module depends on. Investing in clean config management now prevents hardcoded values from leaking into env and training code. This stage creates the skeleton; later stages fill it in.
🗂 Project Scaffold & Config
#1
▸
Create arm_rl/ directory tree
Create: arm_rl/envs/, arm_rl/utils/, arm_rl/configs/, arm_rl/models/, arm_rl/outputs/. Add __init__.py to each Python package so imports work correctly. Keep outputs/ out of version control via .gitignore.
▾
▸
Write ppo.yaml / sac.yaml
Each YAML declares both env params (L1, L2, dt, max_steps, success_threshold, max_vel, action_penalty) and algo params (learning_rate, batch_size, gamma, n_steps for PPO; tau, buffer_size for SAC). Never hardcode these in Python files.
▾
▸
Implement utils/seed.py
Expose set_seed(seed: int) that calls np.random.seed(), torch.manual_seed(), and passes the seed to env.reset(seed=seed). This guarantees bit-for-bit reproducible rollouts — essential for comparing config changes fairly.
▾
▸
Stub utils/logging.py
Wrap Python's standard logging module. Provide get_logger(name: str) that returns a logger with a consistent timestamped format. Used in train.py and eval.py to replace bare print() calls with structured, filterable output.
▾
▸
Stub utils/plotting.py
Provide: plot_training_curves(rewards, distances, path) and plot_trajectory(states, target, path). Centralising matplotlib logic here keeps train.py and render.py clean and makes it easy to swap plotting libraries later.
▾
▸
Write README.md
Document: install steps, quickstart commands (python train.py --config configs/ppo.yaml), config options table, expected outputs, and how to reproduce results. A clear README makes the project reproducible for others from day one — and your future self in 3 months.

Stage 2 — ArmReachEnv (Gymnasium Environment)
The custom environment is the heart of the entire project. It defines what the agent perceives, what it can do, how the arm physically moves, and what signal it gets rewarded on. Every downstream module (training, evaluation, rendering) is a consumer of this env — so correctness here is non-negotiable.
🦾 ArmReachEnv — Gymnasium Environment
#2
▸
State / Observation space
8-dimensional float32 vector: [θ1, θ2, dθ1, dθ2, x_t, y_t, x_e, y_e]. Optionally add [x_t−x_e, y_t−y_e] for 10-dim. Normalize angles to [−π, π]. Declare with Box(low, high, shape=(8,), dtype=np.float32). Correct bounds help with VecNormalize stats.
▾
▸
Action space
Continuous 2D: a = [a1, a2] ∈ [−1, 1]. In velocity control mode scale by max_vel to get rad/s. Declare as Box(low=-1, high=1, shape=(2,), dtype=np.float32). Start with velocity control — it is more stable than torque and easier to debug.
▾
▸
Forward kinematics
Shoulder at origin. Elbow: (L1·cosθ1, L1·sinθ1). Wrist: x_e = L1·cosθ1 + L2·cos(θ1+θ2), y_e = L1·sinθ1 + L2·sin(θ1+θ2). Validate at θ1=θ2=0 → x_e = L1+L2, y_e = 0 before anything else.
▾
▸
Discrete-time dynamics
Per step: dθ += action * max_vel → dθ = clip(dθ, −max_vel, max_vel) → θ += dθ * dt → wrap to [−π, π] via np.arctan2(sin(θ), cos(θ)). Simple Euler integration — no inertia in Phase 1. Default dt=0.02 s.
▾
▸
Reward function
Base: r = −dist where dist = ‖p_e − p_t‖₂. Action penalty: −λ·‖a‖² (λ ≈ 0.01). Success bonus: +R_success (e.g. +10) on reaching. Keep rewards in roughly [−5, +10] range — PPO's value network is sensitive to scale. Consider VecNormalize if reward variance is large.
▾
▸
Termination logic
terminated = dist < success_threshold. truncated = step_count >= max_steps. Return both flags separately per Gymnasium 0.26+ API — step() must return (obs, reward, terminated, truncated, info). SB3 handles bootstrap value differently for each case.
▾
▸
Episode reset
Sample target in reachable annulus: r ∈ [|L1−L2|+0.05, L1+L2−0.05], random angle. Reset joint angles uniformly from U(−0.3, 0.3). Zero velocities. Accept seed kwarg and call super().reset(seed=seed) first for proper RNG seeding.
▾
▸
Info dictionary
Always return: {"distance": float, "success": bool, "end_effector": [x_e, y_e], "is_success": bool}. The is_success alias is read by SB3's EvalCallback to compute and log success rate during training. Don't omit it.
▾
Stage 3 — Environment Verification
Before connecting the env to any training algorithm, run lightweight sanity checks. RL bugs are notoriously hard to trace once training starts — observation shape mismatches, NaN rewards, and Gym API violations cause silent bad learning with no clear error messages. Catch everything here first.
✅ Environment Verification Checks
#3
▸
FK unit assertion
Set θ1=0, θ2=0 and call the FK function directly. Assert |x_e − (L1+L2)| < 1e-6 and |y_e| < 1e-6. Also test θ1=π/2, θ2=0 → x_e≈0, y_e≈L1+L2. Catches sign errors and axis convention bugs before any agent touches the env.
▾
▸
Reset shape & dtype
Call obs, info = env.reset() and assert: obs.shape == env.observation_space.shape, obs.dtype == np.float32, np.isfinite(obs).all(), and all values within declared low/high bounds. Fail fast with a clear assertion error message.
▾
▸
Step NaN stress test
Run 200 random steps with env.action_space.sample(). After each step assert: reward is finite scalar, obs is finite float32, info has required keys. Tests edge cases like angles near ±π wrap and boundary target positions that might produce NaN via div-by-zero.
▾
▸
SB3 check_env() utility
Run from stable_baselines3.common.env_checker import check_env; check_env(env, warn=True). Validates reset/step signatures, space types, info dict format, and warns about common mistakes. Fix all warnings — they become silent errors once you scale to VecEnv.
▾
▸
Reproducibility with fixed seed
Run two identical rollouts using set_seed(42) and assert observation sequences are bit-for-bit identical. This confirms seed.py and env.reset(seed=…) work end-to-end. Required for fair ablation comparisons between configs.
▾
Stage 4 — Training Pipeline (train.py)
Wire the verified environment into an SB3 training loop. This stage is about clean infrastructure: loading configs, logging progress, saving artifacts, and making training resumable. The RL algorithm itself is mostly plug-and-play once the env is correct — the real work here is pipeline plumbing.
🧠 train.py — SB3 Training Pipeline
#4
▸
Load config via YAML + argparse
CLI: python train.py --config configs/ppo.yaml --seed 42 --timesteps 500000. Merge CLI overrides on top of YAML defaults. Log the final resolved config dict at run start and save it as JSON in outputs/ alongside the model — every run is self-documenting.
▾
▸
Wrap env with make_vec_env
Use make_vec_env(lambda: ArmReachEnv(**env_cfg), n_envs=4) for parallel rollout collection. Optionally wrap with VecNormalize(venv, norm_obs=True, norm_reward=True) — this single addition often dramatically speeds up PPO convergence by stabilising the policy gradient magnitude.
▾
▸
Build PPO or SAC model
PPO good defaults: n_steps=2048, batch_size=64, ent_coef=0.01, learning_rate=3e-4. SAC good defaults: learning_starts=1000, tau=0.005, buffer_size=100000. Start with PPO — it's more stable for velocity-controlled tasks and easier to debug via entropy metrics.
▾
▸
Attach EvalCallback
Evaluates on a separate non-vectorised eval env every eval_freq steps and checkpoints the best model. Config: eval_freq=10000, n_eval_episodes=20, deterministic=True, best_model_save_path="outputs/". Reads info["is_success"] to track success rate over training time.
▾
▸
Call model.learn()
Pass total_timesteps, callback=[eval_cb], and tb_log_name="ppo_arm" for TensorBoard. Smoke test: run only 2000 steps to verify no crash. Full training: 200k–1M steps depending on task difficulty and success threshold.
▾
▸
Save model + config + curves
After training: model.save("outputs/final_model"). If VecNormalize was used, also save: vec_env.save("outputs/vec_normalize.pkl"). Call utils/plotting.py to write reward and distance curves. Archive the resolved config JSON too — never lose a run's settings.
▾
▸
Smoke test: 2k steps
Always run a 2000-step train as a CI gate before any long run. If it crashes, the bug is structural (env/model/config mismatch). If rewards are all zero or NaN, the reward function is broken. This saves hours of wasted CPU time on misconfigured long runs.
▾
Stage 5 — Evaluation & Rendering
Quantify and visualise the learned policy. These are kept as separate scripts so evaluation logic is clean and deterministic. eval.py answers "does it work?" with hard numbers. render.py answers "how does it look?" with animation. Both are essential for the Phase 1 definition of done.
📊 eval.py
#5a
▸
Load trained model + env
Use PPO.load("outputs/final_model") and rebuild the env from the saved config JSON. If VecNormalize was used in training, load stats with VecNormalize.load("outputs/vec_normalize.pkl", env) — without this, observations are on the wrong scale and the policy will behave erratically.
▾
▸
Run N evaluation episodes
Roll out 100 episodes in deterministic mode: action, _ = model.predict(obs, deterministic=True). Seed each episode differently for unbiased statistics. Collect per-episode: total reward, final distance, success flag, episode length. Don't reuse the training seed.
▾
▸
Report KPI metrics
Print and save: mean ± std of reward, success rate (%), mean final distance to target, mean episode length. These four numbers are the Phase 1 KPIs. Write to outputs/eval_results.csv for easy tracking and comparison across multiple runs.
▾
▸
Compare vs. random baseline
Run the same N episodes with env.action_space.sample() for random actions. Compute identical metrics. The trained policy must beat the random baseline on all four KPIs for Phase 1 to be declared complete. Include both in the output CSV for clear comparison.
▾
▸
Save CSV + summary plot
Call utils/plotting.py to write a reward distribution histogram and distance-over-episodes line chart. Include a side-by-side bar chart comparing policy vs baseline on each KPI. Save to outputs/eval_summary.png.
▾
🎬 render.py
#5b
▸
Load model + env
Identical loading logic to eval.py — consider extracting to a shared utils/load.py helper. Run in deterministic policy mode for cleaner, repeatable animation. Use a non-vectorised ArmReachEnv directly for easier state access.
▾
▸
Record trajectory states
Step through one full episode storing each frame's [θ1, θ2], end-effector (x_e, y_e), and target position. This list drives the animation without re-running the env during rendering, which keeps frame generation fast and deterministic.
▾
▸
Matplotlib arm animation
Use FuncAnimation. Per frame: draw shoulder→elbow line (upper arm), elbow→wrist line (forearm), dots at joints. Overlay a fading wrist trajectory trail. Keep the plot square with equal axes and a reachable-workspace circle for context.
▾
▸
Show target + success flash
Draw target as a red star marker. When success=True in a frame, switch the marker to green and overlay a "✓ REACHED" text annotation. This immediate visual feedback makes it obvious whether the policy has learned the reaching task at a glance.
▾
▸
Save as .mp4 or .gif
anim.save("outputs/arm_reach.mp4", writer="ffmpeg", fps=30) or use writer="pillow" for GIF with no system dependency. Provide CLI flag --format gif|mp4. Default to GIF for portability; use mp4 for higher quality long episodes.

Stage 6 — Debug Loop
If the policy fails to learn (flat reward curves, zero success rate after 200k+ steps), follow this structured checklist. RL training failures are almost always caused by one of these root causes. Work through them in order from most to least common before changing algorithm or architecture.
🔍 Debug Loop (policy fails to learn)
#6
▸
Check obs/reward for NaN
Temporarily add assert np.isfinite(obs).all() and assert np.isfinite(reward) inside step(). NaN observations silently corrupt neural network weights — the policy learns to output near-zero actions and never recovers. This is the most common silent killer in custom RL envs.
▾
▸
Reward shaping tuning
Plot reward distribution over 1000 random episodes first. If mean reward is always near −5 with little variance, the signal is too weak. Increase success bonus, reduce action penalty λ, or scale distance by a constant. The agent needs clearly distinguishable good vs bad signal to learn from.
▾
▸
Gym compliance re-check
Re-run check_env(env, warn=True) after any env modification. Common violations: wrong obs dtype after a recent change, accidental mutation of the observation array, or not resetting all state variables on reset(). These are invisible unless you actively check.
▾
▸
Hyperparameter sweep
PPO: try learning_rate=1e-4 (more stable), ent_coef=0.05 (more exploration), n_steps=4096 (better gradient estimates). SAC: try learning_starts=5000, larger buffer_size. Log all runs to TensorBoard for fair comparison. Change one parameter at a time.
▾
▸
Normalise observations
Wrap with VecNormalize(venv, norm_obs=True, norm_reward=True). This is often the single biggest fix for PPO — large-magnitude observations cause unstable gradient updates. Save normalization stats after training and load them in eval.py/render.py. Without this step eval will be broken.
▾
▸
Switch action control mode
If velocity control produces oscillating/chaotic trajectories, add damping: dθ *= 0.95 per step. If switching to torque control, reduce max_torque significantly and add velocity damping. Position control (direct angle targets) is the most stable but least physical — use as last resort.

Stage 7 — Final Project File Structure
The completed layout of all files and modules. Each file has one clear responsibility. Mixing concerns — training logic inside the env, plotting inside train.py — makes debugging exponentially harder. This structure ensures that any future change can be localised to a single file.
📁 Project File Structure #7
arm_rl/
├── envs/arm_reach_env.py ← ArmReachEnv class (Stage 2)
├── train.py ← SB3 training entrypoint (Stage 4)
├── eval.py ← Evaluation + metrics (Stage 5a)
├── render.py ← Matplotlib animation (Stage 5b)
├── configs/
│ ├── ppo.yaml ← PPO hyperparams + env config
│ └── sac.yaml ← SAC hyperparams + env config
├── utils/
│ ├── seed.py ← set_seed() reproducibility helper
│ ├── plotting.py ← plot_curves(), plot_trajectory()
│ └── logging.py ← get_logger() wrapper
├── models/ ← (optional custom policy networks)
├── outputs/ ← saved models, plots, eval CSVs, .mp4
├── requirements.txt ← pinned dependencies (Stage 0)
└── README.md ← quickstart, config reference
